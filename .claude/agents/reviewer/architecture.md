# Архитектурные правила проекта

Next.js 14 (App Router), TypeScript, TailwindCSS, статический экспорт (`output: 'export'`).

## Структура

```
src/
├── app/                ← App Router: страницы (page.tsx), layout.tsx, metadata
│   ├── layout.tsx      ← корневой layout, <html lang="ru">, общие метаданные
│   ├── page.tsx        ← главная
│   └── (раздел)/       ← группы роутов по мере роста (события, билеты, ...)
├── components/
│   ├── ui/             ← переиспользуемые примитивы (Button, Card, Container)
│   └── sections/       ← крупные секции страниц (Hero, Afisha, Calendar, Footer)
├── lib/                ← утилиты, хелперы, типы (cn(), форматирование дат)
└── content/            ← данные афиши/событий (пока статичные TS/JSON)
public/                 ← ассеты: картинки, og-image, favicon, manifest
```

## Правила

### Server vs Client компоненты

- По умолчанию компоненты — **серверные** (рендерятся в статику при сборке).
- `'use client'` добавляй **только** при необходимости интерактивности (состояние,
  обработчики событий, хуки браузера). Статичные секции держи серверными.

### Совместимость со статическим экспортом

В `output: 'export'` НЕТ рантайма Node на сервере. Поэтому НЕЛЬЗЯ:

```
app/api/**/route.ts        ← Route Handlers (нет сервера)
'use server'               ← Server Actions
export const dynamic = 'force-dynamic'
import { cookies } from 'next/headers'
```

Динамические роуты обязаны статически генерироваться:

```tsx
// app/events/[slug]/page.tsx
export function generateStaticParams() {
  return events.map((e) => ({ slug: e.slug }));  // из src/content/
}
```

### Данные — отдельно от разметки

Контент (события, афиша, тексты) держим в `src/content/`, компоненты его импортируют.
Это упростит будущий переход на динамику (билеты, запись на мероприятия через API).

```tsx
// НЕЛЬЗЯ: хардкод в JSX
<h2>Концерт 5 июля</h2>

// МОЖНО: данные из content/
import { events } from "@/content/events";
{events.map((e) => <EventCard key={e.id} event={e} />)}
```

### Стилизация

- Только TailwindCSS, без отдельных CSS-файлов на компонент (кроме `globals.css`).
- Хелпер `cn()` (clsx + tailwind-merge) в `src/lib/` для условных классов.
- Адаптивность через брейкпоинты Tailwind, mobile-first.

### Импорты

- Алиас `@/` → `src/` (настроить в `tsconfig.json`).
- UI-примитивы переиспользуются между секциями; секции не дублируют разметку.

## На будущее (когда появится динамика)

Запись на мероприятия и билеты потребуют бэкенда/API. Тогда либо:
- выносим динамику в отдельный сервис (API) + статический фронт обращается к нему, либо
- переводим часть сайта с `output: 'export'` на серверный рендеринг (Node на VDS).

Это осознанный будущий шаг — не тащить серверные фичи в статику преждевременно.
